# Dynamic Lead-rooted Groups; spawn via `omni spawn`, not native subagents

A **Group** is formed at runtime by a **Lead** Session spawning member Agents —
there are no pre-authored rooms or on-disk briefs. To get **separate,
interactive Tiles** per agent, the Lead delegates through the `omni spawn` CLI
(which launches distinct `claude` processes omni places into Tiles), **not** the
native Task/subagent tool (whose subagents run inside the Lead's process and so
cannot be their own Tiles). The Lead decides each agent's role, brief and
working directory on the fly; `omni spawn` gains a `--dir` so a Group can span
repos (e.g. frontend and backend agents in different projects).

## Status

Accepted. **Replaces SPEC decision 4** (explicit pre-authored Room) **and
reframes decision 8** (folder-of-briefs is no longer the launch artifact —
briefs are generated by the Lead at runtime).

## Considered options

- **Native subagents (Task tool)** — Claude spawns them itself, but they are
  internal to one process: one Tile, no separate interactive sub-tiles. Rejected
  because it contradicts the multi-tile vision.
- **Agent Teams (native, experimental)** — spawns separate instances, but wants
  to own tmux, is one-team-per-session and unstable; conflicts with omni owning
  the Zellij layout. Rejected.

## Consequences

- The Lead must be steered to prefer `omni spawn` over the native Task tool — a
  behavioural dependency (injected guidance at launch) that needs validation.
- Two independently launched top-level Sessions never auto-group, even on the
  same Project; collaboration only comes from a Lead spawning members.
